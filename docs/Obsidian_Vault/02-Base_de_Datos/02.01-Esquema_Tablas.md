# Esquema de Tablas (Supabase)

Esta es la descripción de la estructura principal de la base de datos de PostgreSQL (Supabase) del proyecto. El esquema completo tipado para TypeScript se encuentra en `src/integrations/supabase/types.ts`.

## 🧑‍🤝‍🧑 CORE: Agentes (Residentes) e Infraestructura
*   **`datos_personales`**: La tabla maestra de residentes.
    *   **PK:** `id_agente`
    *   **Columnas Clave:** `nombre`, `apellido`, `activo`, `cohorte`, `horas_diarias`.
*   **`dispositivos`**: Los espacios físicos o estaciones de trabajo en el Molino.
    *   **PK:** `id_dispositivo`
    *   **Columnas Clave:** `nombre_dispositivo`, `piso_dispositivo` (1=Papel, 2=Madera, 3=Textil), `cupo_minimo`, `cupo_optimo`, `activo`.
    *   *Nota:* El ID `999` está reservado para el "Descanso / Pool Base".
*   **`turnos`**: Tipología de horarios de las fábricas.
    *   **PK**: `id_turno`
    *   **Columnas Clave**: `tipo_turno` (Mañana=3, Tarde=4, Apertura=45, Descanso=0).

## 📅 PLANIFICACIÓN ESTRATÉGICA (Mes a mes)
*   **`planificacion` / `convocatoria`**: 
    Se planifican los días en que un residente debe asistir, *independientemente de qué dispositivo ocupará*. `planificacion` es el bloque, `convocatoria` es el link 1-a-1 de asistencia teórica.

## ⚙️ CONFIGURACIÓN (Opciones de Días)
*   **`configuracion_turnos`**: Registra la modalidad de trabajo de cada día y turno entero a nivel global.
    *   **PK Compound:** `(fecha, id_turno)`
    *   **Columnas:** `tipo_organizacion` (Valores: 'dispositivos fijos', 'rotacion simple', 'rotacion completa').
*   **`calendario_dispositivos`**: Establece cuántos residentes (el cupo dinámico) se requieren para un dispositivo específico en un día y turno dado. `cupo_objetivo`.

## 🎯 ASIGNACIÓN EJECUTIVA (El Menú del Día)
*Estas tablas mapean qué residente está exactamente en qué dispositivo en un día puntual. Son el motor del Frontend Grid.*
*   **`menu_semana`** (Usada históricamente para Turno Mañana y Tarde).
    *   **Relaciones:** `id_agente`, `id_dispositivo`, `id_turno`, `fecha_asignacion`.
    *   **Columnas Clave:** `numero_grupo` (para manejo de rotación Completa - Grupos A, B, etc).
*   **`menu`** (Usada principalmente para Apertura de fines de semana).

## 🎓 CAPACITACIONES
Para que un residente pueda operar un `dispositivo`, debe estar capacitado en el mismo.
*   **`capacitaciones`**: Instancia de curso. (`id_cap`)
*   **`capacitaciones_dispositivos`**: A qué dispositivos habilita este curso.
*   **`capacitaciones_participantes`**: Asistencia real del agente. `asistio = true`.
*(El frontend junta esto en un Map y genera advertencias (⚠️) si asignamos un agente al piso equivocado).*

## ⚠️ NOTAS IMPORTANTES SOBRE POSTGREST
1.  **Limites Paginación Silenciosos:** Si haces `supabase.from('x').select('*')`, postgREST ignorará el request ilimitado y por seguridad del servidor retornará **únicamente los primeros 1000 registros** históricos (arruinando lecturas mensuales si hay miles de filas). SIEMPRE se debe usar `.eq()`, `.gte()`, `.in()` para filtrar los datos exactos del mes desde la base de datos.
2.  **R L S (Row Level Security):** Si creas una nueva tabla, no olvides correr `GRANT SELECT, INSERT, UPDATE, DELETE ON tu_tabla TO authenticated, anon;` o de los contrario la API escupirá errores HTTP 400 extraños.
